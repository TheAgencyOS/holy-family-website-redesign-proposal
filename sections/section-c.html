Section C

How it is built

Every technology choice in this section answers one question: can Rick Mitchell's team operate, extend, and defend this site for five years without an agency on retainer. The stack is WordPress 6.x on Pantheon, a custom block theme, ACF Pro for structured content, Gravity Forms for Slate, and accessibility engineered in during design and development, not audited in at the end. We name versions, name vendors, and document the boundaries.

C.1

CMS recommendation: WordPress

We recommend traditional, full-stack WordPress as the content management system for the redesigned Holy Family University site. Not headless WordPress. Not a decoupled React or Next.js front end. The same WordPress that powers roughly 43 percent of the web and the majority of private universities that have migrated off Drupal in the past five years. This recommendation is grounded in three factors specific to an institution of more than 3,600 students with a lean web team.

1. Editorial usability for a small team. Rick Mitchell runs a two-person web practice. The CMS has to earn its place for them, not against them. WordPress ships with the Gutenberg block editor, real-time preview, in-context editing, and a media library that editors actually use without training wheels. Drupal's editorial surface has improved in versions 10 and 11, but WordPress is still the benchmark for content author experience, and the gap widens when you measure it in "hours saved per publish."

2. The plugin and developer ecosystem is unmatched. WordPress has the largest active developer community of any CMS, which matters for the practical moment when the university needs to hire a developer, scope a specific integration, or hand off a bug to a freelancer. WordPress talent is abundant and competitively priced. Drupal talent is thinner, pricier, and concentrated in a shrinking number of shops. Across a five-year ownership horizon, that difference compounds into real dollars and real scheduling relief.

3. Lower total cost of ownership over 5 years. WordPress core is free. The premium tools this project needs (ACF Pro, Gravity Forms Elite, Rank Math Pro, a managed Pantheon plan) total under $1,000 per year in licensing. Drupal's modules are also free, but the development and maintenance labor to reach equivalent functionality is consistently higher, as is the cost of every future Drupal major version upgrade. For an institution operating under fiscal discipline (S and P A minus, stable outlook), WordPress delivers more capability per dollar.

Why not headless

Four concrete reasons we do not recommend headless WordPress for the university
  1. Editorial workflow degrades. In a headless setup the preview, in-context editing, and WYSIWYG experience that make WordPress intuitive all require custom re-engineering. Your editors lose the ability to see what a page looks like before publishing unless we rebuild that capability from scratch, on your budget, and maintain it forever.
  2. The plugin ecosystem breaks. Gravity Forms, Rank Math, accessibility checkers, GTM integration, block patterns, and every future plugin you might want all assume they render the page themselves. In a headless architecture, each one needs a front-end reimplementation or a replacement, and the "works on day one" list shrinks to nothing.
  3. It creates a permanent skills gap. A headless front end requires ongoing React or Next.js expertise. Your internal strength is WordPress, PHP, and content. Headless introduces a framework your team cannot maintain independently, which permanently re-ties you to an agency. That is the opposite of the outcome you asked for.
  4. Two codebases instead of one. Headless means maintaining a WordPress back end plus a separate front-end application, each with its own dependencies, deploy pipelines, and security updates. For a site that needs to be stable for five years on a lean budget, this doubles the surface area for bugs and technical debt.

Future-proofing without headless. We model content in Advanced Custom Fields (ACF) Pro with full REST API exposure from day one. Every custom post type, every field group, every taxonomy is reachable over the WordPress REST API. If the university ever needs to serve content to a mobile app, a digital signage system, an admissions kiosk, or a future decoupled front end, the content layer is already an API. You get the editorial experience of traditional WordPress today and the read-API flexibility of headless the moment you need it, without paying for headless up front.

Component architecture

The theme is a custom block theme (not a child theme of a commercial theme, not a page builder theme), built for the university's content types and brand system. Rather than designing pages as monolithic templates, we ship a library of reusable components: hero blocks, program cards, faculty profiles, stat rows, testimonials, event listings, FAQ accordions, CTA patterns. Editors assemble pages from these components inside the block editor. Each component is self-contained (its own markup, styles, and behavior), responsive (designed mobile-first, tested at every breakpoint), accessible (semantic HTML, ARIA, keyboard support, and focus management built in from the first line of code), and documented (a living pattern library ships with the theme, with variants, usage notes, and content requirements). The design system lives in a single theme.json file, which locks the color palette, typography scale, spacing scale, and layout widths. When 15 department content owners publish 400 new pages in year three, they cannot drift off brand, because the editor will not let them.

C.3

Hosting and infrastructure: Pantheon

Holy Family University already runs on Pantheon. We recommend staying there, not as a default, but because Pantheon's specific features address the operational realities of a lean team running a 1,600 URL site. The workflow below is not a suggestion, it is how the platform works.

  1. 1

    Dev

    The working environment. Developers write and test theme and plugin changes here, against a clone of production data. Every code change is a Git commit, tracked, attributed, reversible.

  2. 2

    Test

    A staging environment that mirrors the Live database and files. Code is promoted from Dev to Test for review: this is where Rick's team previews changes, runs accessibility audits, and approves updates before they reach the public site.

  3. 3

    Live

    Production, sitting behind Pantheon's Global CDN on Fastly. Code is promoted from Test to Live only after approval. The Live filesystem is read-only, which means no one (us, you, or an attacker) can make ad hoc edits on production.

Pantheon enforces a deliberate separation: code moves up (Dev to Test to Live, each promotion a Git deployment with a full audit trail) and content moves down (the database and files can be cloned from Live to Test or Dev so developers always work against real content, not placeholder data). If a promotion introduces a problem, one-click rollback returns Live to the previous commit.

C.4

Accessibility compliance: WCAG 2.2 AA

The legal landscape. The ADA Title II Final Rule, published in April 2024, required public universities to meet WCAG 2.1 AA by April 24, 2026. That deadline has passed. As a recipient of federal funds, Holy Family University is independently subject to Section 504 and Section 508, which impose the same standard. We build to WCAG 2.2 AA, one version ahead of the legal floor. WCAG 2.2 is fully backward-compatible with 2.1, so meeting 2.2 automatically satisfies the current legal requirement and provides a compliance margin for the next regulatory update.

The six new success criteria introduced in WCAG 2.2 and the implementation approach for each
Criterion Requirement How we address it
2.4.11 Focus Not Obscured (AA) Keyboard focus must not be hidden behind sticky headers, modals, or other overlapping elements. Focus management designed into every component. Sticky header includes a focus offset; dialogs use scroll padding and scroll-margin-top so the focused element is never obscured.
2.5.7 Dragging Movements (AA) Any drag interaction must have a single-pointer alternative (click or tap). No drag-only interactions in the design. Sliders, reorder controls, and map interactions all expose standard click, tap, and keyboard paths.
2.5.8 Target Size Minimum (AA) Interactive targets must be at least 24 by 24 CSS pixels. Component library enforces a 44 by 44 pixel minimum touch target, which exceeds the 24 pixel floor and satisfies Apple HIG and WCAG 2.5.5 AAA as well.
3.3.8 Accessible Authentication (AA) Login cannot require cognitive function tests (puzzle CAPTCHAs); must allow password managers and paste. All forms support autofill and paste. Spam protection uses honeypot fields and reCAPTCHA v3 (score-based, invisible), never puzzle CAPTCHAs.
3.2.6 Consistent Help (A) Help contact links appear in the same location across pages. The global header and footer include a persistent "Contact" and "Help" link set, enforced by a shared template part rendered on every page.
3.3.7 Redundant Entry (A) Do not require users to re-enter information they have already provided in the same process. Multi-step Gravity Forms carry values forward between steps. The Request for Information flow auto-populates known Slate Ping cookie data where consented.

Shift left. Most agencies treat accessibility as a QA checkpoint: build first, audit later, fix under deadline pressure. That is how sites ship with 200 open violations. We shift accessibility left into every phase. Design: color contrast validated before a pixel is placed, touch targets sized in wireframes, focus states designed as a first-class visual. Development: semantic HTML and ARIA from the first line of code, axe-core linting in the IDE and in the CI pipeline, pull requests that introduce regressions cannot merge. Content: editor guardrails enforce alt text, heading hierarchy, and link purpose. QA: manual keyboard-only and screen reader testing with NVDA, JAWS, and VoiceOver on the templates that automated tools cannot validate. Accessibility is not a phase; it runs alongside everything else.

On the Lighthouse Accessibility score of 96 for the university's homepage

A Lighthouse Accessibility score of 96 reflects only automated axe-core checks, which catch 30 to 57 percent of WCAG failures. The real gap includes manual test categories like focus order, labels for custom controls, and reading order. The homepage scan does not exercise program pages, the 23 request-for-information forms, or Slate integration surface area. Our shift-left methodology addresses the axe-covered 30 percent AND the manual 70 percent, on every template.

Source: axe-core documentation, Deque Systems; Lighthouse 12.8.2 homepage scan 2026-04-10.

Common university website failures we prevent

  • Missing alt text (WCAG 1.1.1). Enforced at the editor level. Images cannot be inserted without an alt value (or an explicit "decorative" flag that writes alt="").
  • Insufficient color contrast (WCAG 1.4.3). Palette validated in the design system to 4.5:1 for body text and 3:1 for large text and UI components before a pixel ships.
  • Keyboard navigation failures (WCAG 2.1.1). Every interactive component tested with keyboard only. No interaction is mouse exclusive.
  • Missing video captions (WCAG 1.2.2). Wistia (already in your stack) supports captions natively; captioning is a content requirement in the governance doc, not a nice-to-have.
  • Inaccessible PDFs (WCAG 1.3.1). Accessible PDF templates provided for new documents; legacy PDF remediation scoped as a post-launch phase, not buried in launch scope.
  • Form labeling failures (WCAG 3.3.2). Every Gravity Forms field carries a programmatic label, explicit error messaging, and instructions. No placeholder-as-label patterns.
C.5

Integration capabilities

Slate CRM integration

Slate is Holy Family's enrollment CRM. A successful redesign has to integrate with Slate cleanly at three levels: form capture, behavioral tracking, and dynamic content. We use native WordPress patterns for each, not raw Slate script embeds.

Pattern 1

Forms via Gravity Forms webhook

We build native Gravity Forms that match the site's design system exactly (typography, color, spacing, validation). The Gravity Forms Webhooks Add-On POSTs each submission as JSON to the Slate Source Format endpoint in real time, with field keys mapped per form. No <script> embed, no WordPress sanitizer conflicts, no CSS collisions, no per-page form limit. This is the pattern documented end-to-end by NC State University, including complete field key mappings between Gravity Forms and Slate.

Real time to Slate
Pattern 2

Behavioral tracking via Slate Ping

Slate Ping is a lightweight JavaScript snippet (served from technolutions.net/ping) that tracks page-level browsing, drops a first-party cookie, and links anonymous visits to known prospect records the moment a visitor performs an authenticated action. We deploy it site-wide via the theme header or, if preferred, through the existing GTM container. Every page is tracked from launch day.

Enrollment attribution
Pattern 3

Dynamic content via Slate Query Builder

For content that originates in Slate (program listings, admissions events, application deadlines), we pull cached JSON from Slate's Query Builder web service into WordPress. The website always reflects current enrollment data without manual content duplication. Cache TTLs are configurable per query so admissions can make a change in Slate and see it on the website within minutes, not at the next overnight batch.
